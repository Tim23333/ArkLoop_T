# ArkLoop 使用说明

ArkLoop 是面向《明日方舟》MuMu 模拟器 12 的作战时间轴录制、编辑与精确回放工具。

当前版本只支持实时桌面流程：

```text
MuMu 实时画面与鼠标操作
  -> ArkLoop 识别部署 / 技能 / 撤退
  -> 外部 WebSocket 提供绝对 frame_count
  -> 生成并编辑帧时间轴
  -> PlaybackController 精确暂停并回放
```

费用条校准、费用像素计时、离线视频识别、Excel 执行和旧 CLI 已经退出当前运行逻辑。

## 一、运行前准备

### 1. 系统与模拟器

- Windows 10/11 64 位。
- MuMu 模拟器 12，直接启动目标实例。
- 建议将游戏分辨率设为 **1280×720**。
- 建议使用 **DirectX** 渲染。
- 录制期间不要移动、缩放或最小化 MuMu 窗口。

旧的 `PRTS+键鼠方案示例.json` 不再是精确回放的必需配置。当前回放会直接点击游戏右上角的暂停按钮，并通过图像识别确认暂停或恢复状态。

### 2. 外部游戏时间服务

ArkLoop 不再从费用条推断帧数。录制和回放必须连接一个持续推送游戏绝对帧的 WebSocket 服务。

消息格式：

```json
{
  "game_time": 12.345,
  "frame_count": 185,
  "connected": true
}
```

字段含义：

| 字段 | 作用 |
| --- | --- |
| `frame_count` | 本次战斗从 0 开始的绝对逻辑帧，是录制和回放的唯一调度依据 |
| `game_time` | 用于界面显示的游戏时间 |
| `connected` | 时间服务是否成功读取游戏数据 |

默认地址为 `ws://127.0.0.1:59555`。外部时间服务需要单独启动，本仓库不包含该服务。

### 3. 启动 ArkLoop

使用打包版本时运行：

```text
ArkLoop.exe
```

打包版本默认使用 CPU 头像识别，主程序中不包含体积较大的 Torch/CUDA。可直接点击主页面下方的 CPU/GPU 模式按钮尝试运行时切换。首次安装或修复 GPU 依赖时，运行同目录下的：

```text
ArkLoopDependencyInstaller.exe
```

- 选择“仅使用 CPU”时，只保存 CPU 模式，不会下载任何额外依赖。
- 选择“使用 GPU 加速”时，安装程序先检查当前目录；依赖版本正确且文件完整时直接启用，不会重复下载。只有缺失、损坏或版本不符时才下载 CUDA 12.1 版 PyTorch。安装过程建议预留至少 12 GB 磁盘空间。
- GPU 依赖保存在 `dependencies/gpu/site-packages`，不会写入系统 Python，也不会放到 C 盘缓存目录；安装程序使用无缓存下载方式，临时文件由系统安装流程在结束后清理。
- 2.45 GB 的 Torch wheel 支持断线续传。下载中断时，已完成部分保存在 `dependencies/downloads/*.part`；重新运行安装器并选择 GPU 后会继续下载，安装成功后自动删除。
- 安装完成后回到 ArkLoop，再次点击 CPU/GPU 模式按钮即可切换，无需重启。
- GPU 依赖加载失败或 CUDA 不可用时，ArkLoop 会自动回退到 CPU 识别。

使用源码版本时运行：

```powershell
.venv\Scripts\python.exe scripts\arkloop_webview.py
```

源码安装与构建步骤见 [README.md](./README.md)。

## 二、首次配置

启动后点击左上角设置按钮。

### MuMu 设置

| 设置项 | 填写说明 |
| --- | --- |
| MuMu 模拟器安装目录 | MuMu Player 12 的根目录，其中应包含 `nx_device`、`nx_main` 或 `shell` 等目录 |
| MuMu 父窗口标题 | 通常为 `MuMu模拟器12`，必须与当前窗口标题一致 |
| 渲染子窗口标题 | 通常为 `MuMuPlayer`；多开或不同版本可能为 `MuMuPlayer1` 等 |
| 实例索引 | 主实例通常为 `0`，多开时按实际实例填写 |
| 截图后端 | 推荐 `auto`；DLL 截图失败时会回退到 Win32 |

窗口标题现在采用精确匹配。找不到渲染子窗口时，可以运行仓库中的 `list_mumu_windows.cmd`，或使用设置窗口给出的检测命令查看真实标题。

修改 MuMu 安装目录后应重启 ArkLoop，因为 MuMu DLL 和截图句柄会在首次使用时初始化。

### 时间源设置

1. 在“时间源 WebSocket 地址”中填写服务地址。
2. 点击“测试连接”。
3. 确认状态显示“已连接”。
4. 如果显示“已连接（内存未就绪）”，说明 WebSocket 已连接，但服务尚未正确读取游戏。

时间服务正常时，主界面工具栏会持续显示“游戏时间”和“帧数”。如果显示“未连接”或 `--`，不要开始录制或回放。

## 三、界面与控制按钮

时间轴左上角有四个运行按钮：

| 按钮 | 作用 |
| --- | --- |
| 录制 | 开始实时记录 MuMu 操作 |
| 停止 | 结束当前录制或彻底停止当前回放 |
| 播放 | 从当前续录偏移开始执行选中的时间轴 |
| 暂停 | 暂停当前录制或回放，并保留当前位置用于续录或续播 |

暂停和停止的语义不同：

- **回放暂停**：控制器让游戏保持暂停，并保留当前绝对帧、已部署干员状态和续录偏移。
- **录制暂停**：停止本段录制、追加已识别动作并保存续录偏移，但不会主动暂停游戏画面。
- **停止**：放弃本次续播状态，将续录偏移重置为 `0`；如果游戏正处于程序控制的暂停状态，会先恢复游戏。

空闲时工具栏会显示“续录偏移”。它表示下一次录制或播放从原时间轴的哪个绝对帧继续。

工具栏中的“同步资源”会一键检查并更新最新干员头像、关卡格子结构 JSON 及其必要索引。同步只在未录制、未播放时可用；下载默认读取 Windows 系统代理，只获取新增或变化文件。全部下载和校验完成后程序才会替换资源，网络中断不会破坏当前可用版本。打包版会把同步结果保存在 `ArkLoop.exe` 同目录的 `resource` 文件夹，后续启动优先使用该版本。

## 四、新建时间轴

1. 点击侧边栏的 `new TimeLine`。
2. 搜索并选择关卡代号或名称。
3. `max_tick` 保持默认值即可。它只用于加载旧 `cycle/tick` 时间轴，不控制当前精确回放。
4. 如果关卡内已有可视作干员的装置，可填写装置名称和地图坐标，例如 `C3`。
5. 点击“创建”。

常用关卡和装置配置可以保存为预设。预设只保存时间轴设置，不包含动作。

新时间轴中的动作使用绝对 `frame`，可执行类型只有：

- `部署`
- `技能`
- `撤退`

## 五、录制时间轴

### 新时间轴录制

1. 启动 MuMu 和外部时间服务。
2. 进入对应关卡，确认界面帧数正在变化。
3. 在 ArkLoop 中选中刚创建的空时间轴。
4. 点击录制按钮。
5. 将焦点切回 MuMu，再进行部署、技能和撤退操作。
6. 点击停止按钮结束录制。
7. 在保存窗口中确认时间轴名称。

录制时 ArkLoop 会实时监听鼠标点击和拖拽，结合 MuMu 画面识别动作语义，并把当时的 WS `frame_count` 写入动作。

### 部署录制

- 从底部干员栏正常拖动到目标格。
- 松手位置尽量靠近格子中心，避免落在相邻格判定边缘。
- 按游戏正常方式拖动部署方向。
- 如果头像识别失败，时间轴中可能出现错误干员或空缺，需要在编辑器中修正。

### 技能与撤退录制

- 先正常选中场上干员，再点击技能或撤退。
- 技能未就绪时不要反复点击，否则无效点击仍可能进入识别流程。
- 不要选中被详情面板遮挡或无法正常响应的干员。

### 暂停录制

录制中点击 ArkLoop 的暂停按钮会：

1. 停止本段输入监听。
2. 将本段新动作追加到当前时间轴。
3. 保存当前绝对帧为续录偏移。

当前录制暂停不会依赖旧费用条逻辑，也不会使用 JSON 中的 `bullet_threshold` 或 `frame_threshold`。

录制暂停不会自动点击游戏内暂停按钮。如果还需要在 MuMu 中停住战斗，请自行操作游戏暂停；再次录制前确认帧数状态符合预期。当前录制续接主要保证动作追加和绝对帧偏移，复杂场面下仍应检查已部署干员的后续技能、撤退识别是否正确。

## 六、回放时间轴

### 推荐启动方式

1. 选中时间轴，准备与时间轴一致的队伍和关卡。
2. 确认 WS 状态已连接，且 `frame_count` 会在战斗开始后从 0 重新计数。
3. 点击 ArkLoop 的播放按钮。
4. 手动开始战斗，保持游戏正常运行。

> 不要按照旧说明在开始播放前把游戏一直保持在暂停状态。当前调度需要帧数继续前进，程序会在每个动作前自行进入精确暂停。

如果 WS 仍保留上一场战斗的较大帧数，播放线程会等待新战斗的帧数重置到 `0` 或 `1`。

### 每个动作的实际执行逻辑

对部署、技能、撤退三个动作，控制器都会执行同一套精确到帧的前置流程：

1. **等待目标前 30 帧**：正常等待 WS 帧数前进。
2. **动作预选**：部署会预选底部待部署干员区域；技能和撤退会预选场上干员。
3. **等待目标前 10 帧**：继续读取绝对帧。
4. **进入暂停**：点击游戏暂停按钮，并通过暂停图标的图像识别确认游戏确实暂停。
5. **逐帧推进**：短暂恢复，等到 WS 帧号实际前进后立即再次暂停；每一步都重新确认暂停图标。
6. **到达目标帧**：在暂停状态执行部署、技能或撤退。
7. **恢复游戏**：等待动作输入稳定后点击继续，并通过图像识别确认游戏已经恢复；第一次点击被吞时会自动重试。
8. **等待下一动作**：游戏保持运行，直到下一个动作进入相同流程。

暂停状态判断来自游戏画面的暂停图标，不使用 WebSocket 传来的暂停状态。WebSocket 只负责提供绝对帧号。

### 部署动作的回放顺序

部署在暂停状态下按以下顺序完成：

1. 定位目标干员头像。
2. 按住头像并平滑拖动到目标格。
3. 在目标格松开，完成部署。
4. 等待部署输入稳定。
5. 从目标格再次滑动部署方向并松开。
6. 动作完成后恢复游戏。

也就是说，当前逻辑不是“拖动干员时同时带出方向”，而是先部署并松手，再单独滑动方向。

### 技能与撤退回放

- 技能：预选对应干员，到目标帧后在暂停中点击技能按钮。
- 撤退：预选对应干员，到目标帧后在暂停中点击撤退按钮。
- 两种动作完成后都会确认游戏恢复，再进入下一个动作。

### 迟到动作

如果开始处理动作时，当前帧已经超过目标帧两帧以上，动作会被跳过，并在日志中输出 `Skipping action`。这通常表示：

- 播放开始太晚。
- 游戏在开始播放前已经运行了一段时间。
- 暂停或继续点击未被及时处理。
- WS 帧数据延迟或中断。

## 七、暂停、续播与续录

### 暂停回放

点击暂停按钮后，控制器会确保游戏保持暂停，并保存：

- 当前绝对帧。
- 已部署干员及其地图位置。
- 当前识别状态。
- 续录偏移。

之后可以：

- 再次点击播放，从该偏移继续执行剩余动作。
- 点击录制，从该帧继续录制并追加新动作。

### 完整停止回放

点击停止按钮会清除续播状态和续录偏移。下一次播放从时间轴起点开始。

### 手动修改续录偏移

空闲时可以直接修改工具栏中的“续录偏移”：

- `0`：从时间轴起点开始。
- 大于 `0`：跳过该帧之前的动作，并把后续动作映射到新战斗的相对帧。

自然播放完成后界面可能保留最终帧作为偏移。要重新从头播放，可以重新选择时间轴或手动把偏移改为 `0`。

## 八、断点

停止运行后，在时间轴空白处点击右键并选择“添加断点”。右键已有断点可以删除。

播放到断点时：

1. 控制器暂停游戏。
2. 保存当前帧和运行状态。
3. 结束本段播放。

随后可以继续播放，或者从断点开始录制后续动作。编辑和回放过程中使用的都是绝对帧断点。

## 九、编辑时间轴

运行中的时间轴不能编辑。停止或暂停后可以：

| 操作 | 方法 |
| --- | --- |
| 新增动作 | 双击部署、技能或撤退轨道的空白位置 |
| 编辑动作 | 右键动作块，选择“编辑” |
| 删除动作 | 右键动作块，选择“删除” |
| 调整帧数 | 直接拖动动作块 |
| 添加断点 | 右键时间轴空白位置 |
| 删除断点 | 右键断点标记 |

手动新增部署动作时必须确认干员、地图坐标和方向。技能与撤退动作必须引用此前已经部署或在关卡设置中预置的干员/装置，否则回放时无法确定场上坐标。

## 十、时间轴文件管理

侧边栏支持：

- `Pin / Unpin`：固定或取消固定常用时间轴。
- `Rename`：重命名。
- `Duplicate`：复制一份用于分支尝试。
- `Export`：导出 JSON。
- `Delete`：删除。
- “导入”：从外部导入 JSON，重名时自动增加序号。

时间轴保存在运行目录的 `timelines/` 中：

- 源码运行：项目根目录下的 `timelines/`。
- 打包运行：`ArkLoop.exe` 旁的 `timelines/`。

导入、导出和保存时会自动删除旧的 `bullet_threshold`、`frame_threshold` 设置。

## 十一、地图截图辅助

选中包含有效 `map_code` 的时间轴后，点击上方地图工作区可以截取当前 MuMu 画面并叠加格子坐标。

截图失败时检查：

- MuMu 是否正在运行。
- 父窗口和渲染子窗口标题是否正确。
- 截图后端是否可用。
- 当前时间轴的关卡代号是否存在于资源数据中。

## 十二、常见问题

### GPU 加速没有生效

1. 先点击主页面的 CPU/GPU 模式按钮；若提示依赖缺失，运行 `ArkLoopDependencyInstaller.exe` 并选择“使用 GPU 加速”。
2. 确认程序目录中存在 `dependencies/gpu/site-packages/torch`。
3. 使用 `nvidia-smi` 检查 NVIDIA 驱动是否正常，并确认驱动支持 CUDA 12.1 运行库。
4. 安装完成后再次点击 GPU 模式，检查日志中是否出现 `CUDA available`；出现 `CPU path` 时说明程序已安全回退到 CPU。

CPU 模式不需要运行安装流程。选择“仅使用 CPU”后即使磁盘上保留旧 GPU 文件，主程序也不会加载它们。

### 时间显示“未连接”

1. 确认外部时间服务已经启动。
2. 检查设置中的 WS 地址和端口。
3. 点击“测试连接”。
4. 检查时间服务是否持续发送合法 JSON。
5. 如果显示“内存未就绪”，在游戏进入战斗后检查服务的游戏读取状态。

### 点击播放后一直没有动作

- 确认游戏没有被手动保持暂停。
- 确认战斗已经开始且帧数正在增加。
- 确认第一条动作帧没有早于当前帧。
- 如果上一场帧数仍很大，开始一场新战斗等待帧号重置。

### 日志提示 `Unable to verify game pause`

程序没有从画面识别到暂停图标。检查：

- 分辨率是否为 1280×720。
- 游戏右上角暂停按钮是否被遮挡。
- 截图窗口是否指向正确的 MuMu 渲染子窗口。
- 游戏 UI 缩放或渲染模式是否与识别资源差异过大。

### 日志提示 `Unable to verify game resume`

继续点击后画面仍被识别为暂停。程序会自动重试；持续失败时检查暂停按钮区域、游戏弹窗和 MuMu 输入是否正常。

### 日志提示 `Frame step overshot target`

逐帧过程中一次恢复跨过了多个逻辑帧。请保留完整日志，尤其是相邻的 `Frame step A -> B (target C)` 行，以便判断是 WS 延迟还是暂停点击处理过慢。

### 部署失败或一闪而过

- 确认目标干员在底部部署栏中可见。
- 确认时间轴干员名称与当前编队一致。
- 检查头像模板是否包含当前皮肤。
- 检查费用是否足够；当前执行器不会通过旧费用条识别等待费用。
- 检查目标格是否可部署，以及地图和坐标是否正确。

### 技能或撤退点错位置

技能和撤退通过该干员的已部署坐标定位。确认时间轴中此前存在正确的部署动作，或者已在关卡设置中预置对应装置。

### 录制动作识别错误

- 开始录制后先把焦点切回 MuMu。
- 录制期间不要移动 MuMu 窗口。
- 操作完整、清晰，避免在极短时间内连续误点。
- 在时间轴编辑器中修正错误干员、坐标、方向或帧数。

## 十三、日志与问题反馈

源码入口会输出控制台日志；打包版不会打开 CMD 窗口，日志保存在 `ArkLoop.exe` 同目录的 `logs/arkloop.log`。反馈问题时请提供：

- 从开始播放或录制到问题发生后的完整日志。
- 对应时间轴 JSON。
- MuMu 与 ArkLoop 界面截图。
- WS 服务地址、连接状态和问题发生时的帧数。
